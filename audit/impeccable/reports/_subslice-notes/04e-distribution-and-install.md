## Paradigms Worth Importing

- Model provider identity separately from layout destinations. Impeccable's `codex -> .agents` plus `.codex/hooks.json` sidecar is the clean pattern for YoinkIt's likely "one harness, multiple config locations" reality.
- Copy compiled harness variants from a universal bundle. Do not symlink every target to one shared skill when provider-specific prompt text, script paths, hook manifests, or metadata differ.
- Use a path-normalized content hash for updates. The `.PROVIDER/skills/` normalization lets the updater distinguish true content drift from install-method path differences; YoinkIt can use the same idea for `skill/codex` vs `skill/claude` paths and generated driver adapter names.
- Keep generated output committed only when a consumer path needs raw GitHub/submodule reads. Pair that with source-first PR rules, exact generated-path CI sync, and a release-time rebuild-drift guard. The discipline matters as much as the generator.
- Generate a slim marketplace/plugin subtree. Even after correcting the size, the architectural point holds: a plugin install should not drag the capture engine, extension wrapper, site, fixtures, and audit corpus unless the runtime actually needs them.
- Maintain a dated harness matrix and a count reconciliation section. Impeccable has at least four legitimate provider counts; the reports are readable because each layer's count is named and justified. YoinkIt should do the same before its Codex/Claude/browser-driver matrix grows.
