## Paradigms Worth Importing

- Use a single coordinator for extension state, but make the coordinator disposable. Impeccable's SW is the routing hub, yet the page engine and UI can reconnect and rescan. For YoinkIt, the SW can coordinate capture, but the irreplaceable timing data should live where it can survive SW sleep or be explicitly invalidated.
- Preserve operation identity across the bridge. Impeccable already has an optional `scanId` shape in the engine, but loses it in the content script. YoinkIt should treat `captureId`/`armId` as a first-class field in every message, including errors and ready/ack events.
- Keep two transports: one-shot `sendMessage` for popup-style commands, named ports for long-lived panels with disconnect semantics. This maps cleanly to a YoinkIt popup trigger plus a DevTools Capture panel.
- Reuse the connect-then-heal pattern, but adapt the healing action. Impeccable heals "no findings" by rescanning. YoinkIt may need "capture expired, re-arm required" or "restore page-owned capture ticket" instead of automatic replay.
- Make settings invalidation explicit. Impeccable's `disabled-rules-changed` action is really a config invalidation broadcast. YoinkIt should name this after the domain concept and rebuild capture config from one source of truth before sending the next page command.
