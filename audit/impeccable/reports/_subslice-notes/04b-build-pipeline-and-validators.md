## Paradigms Worth Importing

- Keep the leaf's strongest transfer: cheap structural gates first, fan-out in the middle, content/artifact validators at the end, and thrown artifact failures wired into a non-zero build exit. For YoinkIt this maps cleanly to "malformed capture spec fails before write" and "empty capture artifact fails loudly."
- Import the universal README mismatch as a new YoinkIt warning: whenever a generated bundle contains an inventory, validate that inventory against the provider/config table. The Impeccable source copies 13 provider dirs but narrates only 11 in its README string.
- The root sync stash/restore pattern is directly useful for YoinkIt if it ever regenerates skill copies while preserving project-local capture state. The important part is that the preservation list is explicit and scoped to runtime artifacts, not a broad copy-back.
- The hook manifest/runtime split is clean: one shared runtime script, provider-specific manifests only for event names, matchers, and path resolution. YoinkIt should keep capture logic single-source and fork only invocation manifests if it adds harness hooks.
- The named-export/test nuance is a caution for a future YoinkIt build: if provider configs are table-driven, tests should iterate that same table or intentionally document excluded providers. Partial named helper exports create an attractive but incomplete coverage story.
- The no-op validator is a caution to import negatively. A retained 0-returning gate reads like protection but cannot protect anything; remove obsolete checks or make their no-op status explicit in build output and docs.
