## Deep Implementation Notes

- `command-metadata.json` is exactly the schema the leaf states: every value has only `description` and `argumentHint` (`skill/scripts/command-metadata.json:1-94`). The recomputed unique schema shape is `argumentHint,description`; no `category` key exists.
- Router count is correctly derived from the markdown command table. `SKILL.src.md:121-145` has 23 backtick command rows, and `generateCounts` applies `/^\| \`[^\`]+\` \|/gm` at `scripts/build.js:33-43`. It writes `COMMAND_COUNT`/`DETECTION_COUNT` at `build.js:57-64`, validates five files at `build.js:66-105`, returns errors at `build.js:107-112`, and the build exits non-zero when those errors are present at `build.js:766-783`.
- The count validator does not reconcile router rows with metadata keys. The OG card uses `Object.keys(metadata).length` at `scripts/generate-og-image.js:35-42` and renders the value at `generate-og-image.js:137`; `generateCounts` uses router rows. A router/metadata mismatch can still produce two different public counts.
- `VALID_COMMANDS` is a full 23-command copy, not a reduced pinnable subset (`skill/scripts/pin.mjs:29-35`). The CLI blocks unknown commands before filesystem writes at `pin.mjs:202-206`, and missing-arg usage prints the same list at `pin.mjs:191-195`.
- `IMPECCABLE_SUB_COMMANDS` is the intentional 19-command suggestion subset (`scripts/lib/utils.js:706-712`). The excluded router commands are exactly `craft`, `init`, `extract`, and `live`; the replacement path uses this list for `{{available_commands}}` at `utils.js:714-739`.
- `SKILL_CATEGORIES` in the build-side JS source has the claimed 24 keys, `impeccable` plus 23 commands (`scripts/lib/sub-pages-data.js:47-80`). Its buckets differ from router-table labels: router categories are Build/Evaluate/Refine/Enhance/Fix/Iterate in `SKILL.src.md:123-145`, while category order is create/evaluate/refine/simplify/harden/system at `sub-pages-data.js:80`.
- The pin safety model is confirmed. `PIN_MARKER` is defined at `pin.mjs:37-38`; `pin` skips existing non-marker skills at `pin.mjs:126-135`; `unpin` refuses to remove files without the marker at `pin.mjs:166-173`. This is a strong marker-in-artifact pattern for YoinkIt-generated files.
- `findHarnessDirs` only acts where a harness already has `skills/impeccable` or `skills/i-impeccable` (`pin.mjs:63-74`). This confirms the install-footprint mirroring claim.
- `PROVIDER_PLACEHOLDERS` has one `$` prefix provider, `codex`, at `scripts/lib/utils.js:583-588`; the other listed placeholder entries use `/` (`utils.js:564-631`). `replacePlaceholders` substitutes `{{command_prefix}}` only in markdown/reference/agent transform paths, not script copies (`utils.js:714-739`, `factory.js:228-239`, `factory.js:246-266`).
- Site register split is richer than the leaf says. `scripts/lib/api-data.js:27-70` turns metadata descriptions plus `site/content/skills/<id>.md` taglines into `commands.json`; `site/content/skills` contains 24 files, including `impeccable` plus the 23 commands. This is good evidence for the two-register pattern, but it also means metadata changes affect public API/docs surfaces.

## Paradigms Worth Importing

- Import the marker-in-generated-artifact pattern directly for any YoinkIt generated user-project files. The crucial part is symmetric behavior: create/update only when absent or marker-owned, delete only when marker-owned.
- Import the two-register copy model, but name the consumers in the design: agent-trigger descriptions in one metadata file, human taglines in docs/content, and an explicit generator/API contract that reconciles both.
- Import the count validator with a stronger set-equality companion. For YoinkIt, any "supported drivers", "capture primitives", "harnesses", or "commands" list should have both a derived count and an exact identity comparison across allowlists, docs maps, site data, and generated APIs.
- For runtime generators, route every emit path through the same placeholder resolution or avoid placeholders in emitted artifacts. The pin shim shows the failure mode: build-time markdown transforms do not protect runtime-written scripts.
