## Paradigms Worth Importing

- Import the one-core, thin-adapter shape, with the caveat that "shared core" can still choose different engines by hook posture. For YoinkIt, that means one validation/capture comparison core and small Claude/Codex/Cursor shims, but explicit capability flags for "can run real browser now" versus "can only validate static JSON."
- Import fail-open as a hard product rule. A YoinkIt hook should treat page failures, browser unavailability, malformed specs, and validator exceptions like Impeccable treats detector throws: audit if possible, emit nothing or allow, and never break the agent turn.
- Import event-shape recovery. Codex `apply_patch` proves the hook cannot rely only on structured `file_path`. YoinkIt should parse patch bodies for changed recreation/spec files and then decide whether a capture comparison is warranted.
- Import the emission-priority ladder, not just the output envelope. Fresh drift findings should beat re-nudges; re-nudges should beat clean acks; clean acks should be gated to files where reassurance is useful; stale/suppressed cases should fall silent.
- Import the latency-matched generated-file split. YoinkIt can use cheap path guards in hot hooks, then richer git/header checks in slower interactive capture or map phases.
- Treat pre-write reconstruction as a narrow tool. It is valuable for static, in-memory checks, but YoinkIt's core truth signal is observed motion after a real browser run. The leaf's conclusion that YoinkIt should default to post-edit advisory feedback is correct.
