## Paradigms Worth Importing

- Keep the provider-native manifest builder pure and side-effect free, but have every writer consume that same builder output. Impeccable's build module is the good half; the runtime `hook-admin.mjs` second shape definition is the drift risk YoinkIt should avoid.
- Model the installer as "bundle reader plus merge contract". `copyProviderHooks()` does not synthesize provider JSON from scratch; it reads the generated artifact, strips prior own entries, appends the fresh entry, and preserves user hooks. This is the right posture for YoinkIt because hook installation is a user-project mutation and must be idempotent.
- Steal the local/shared duplicate guard directly. The Claude `settings.local.json` write plus `settings.json` marker check solves the real double-fire hazard without overriding a user's deliberate shared install.
- Preserve the non-interactive consent nuance. Default-on in `-y` or no-TTY mode keeps installs useful, but not writing `accepted` avoids fabricating a remembered human decision. That is especially important for YoinkIt if capture-validation hooks have runtime cost or privacy implications.
- Treat `skills link` as an open product question for YoinkIt. Impeccable's current link path does not install hooks, which may be intentional to keep linking side-effect-light, but a YoinkIt source-link workflow may need an explicit `yoinkit hooks install` follow-up or a documented "link does not wire hooks" message.
